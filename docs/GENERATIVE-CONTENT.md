# Generative Content System

## Design Document (DRAFT v0.1)

---

## 1. The Problem

The game currently has ~375 hand-authored content items across 5 rounds, plus 5 sets of pre-written decisions (individual + team). This content is high quality but static — the same Slack messages, news headlines, and decision prompts appear regardless of what players actually did. The "branching" is limited to `ContentItem.condition` gates (show/hide based on a state threshold), not true reactivity.

This creates two problems:
- **Replayability ceiling.** Players who replay see the same content. The scenario feels scripted rather than emergent.
- **Narrative disconnection.** If OpenBrain chose to go public in Round 1, the Round 2 Slack messages should reference that. If China attempted a weight theft that failed, the Round 3 intelligence briefing should reflect the aftermath. Currently, content can only gate on numeric thresholds — it can't reference *what happened* or *who did what*.

The goal: after Round 1 (which remains pre-authored as the "inciting incident"), all subsequent content — app items, briefings, NPC messages, and decisions themselves — is generated by an LLM, reactive to the actual game state and player choices.

---

## 2. Design Principles

**Skeleton + flesh.** The narrative skeleton (5-round arc, escalation curve, key story beats, threshold events) stays pre-authored. The LLM generates the flesh: specific messages, headlines, memos, and decision framings that drape over that skeleton and react to what actually happened.

**State is the source of truth.** The LLM never invents state. It reads the current `StateVariables` (all 33 of them), the `RoundHistory[]` (decisions made, state deltas), and the `firedThresholds` set. It generates content that *reflects* this state, not content that *contradicts* it.

**Structured output, not free text.** The LLM outputs typed JSON matching the existing `ContentItem`, `DecisionOption`, and `Briefing` schemas. The server validates and injects these into the existing pipeline. Client code doesn't change.

**No free lunches, even in generation.** Generated decisions must have genuine tradeoffs. Every option must advance at least one variable the player cares about while degrading at least one other. The system enforces this structurally.

**Fail gracefully.** If generation fails or is too slow, fall back to pre-authored content for that round. The game never stalls waiting for an LLM.

---

## 3. What Gets Generated vs. What Stays Fixed

### Fixed (pre-authored, all rounds)

| Element | Why it stays fixed |
|---|---|
| **Round 1 content** (all apps) | The inciting incident must be tightly controlled. Sets tone, teaches players what to look for. |
| **State variable definitions** | The engine itself. 33 variables, their ranges, their clamping rules. |
| **Fog of war rules** | Which faction sees what, with what accuracy. Structural, not narrative. |
| **Threshold events** | The 8 once-per-game events (weight theft, auto-leak, board revolt, etc.). These are mechanical — they fire based on state, apply fixed deltas. |
| **Ending arc resolvers** | The 9 ending arcs and their spectrum logic. These are the scoring function. |
| **Phase structure & timing** | 5 phases per round, durations, round count. |
| **Activity penalties** | Primary app requirements per role. |

### Generated (Rounds 2-5)

| Element | What the LLM produces | Constrained by |
|---|---|---|
| **Briefings** | `common` text + `factionVariants` per round | Round arc template, current state, decision history |
| **App content** | `ContentItem[]` per faction per app | App schema, content budget (3-5 critical, 5-10 context, 1-2 red herrings, 1-2 breadcrumbs), faction voice |
| **Decisions** | Individual (`role → prompt + 3 options`) and team (`faction → prompt + 3 options`) | Effect budget constraints, no-free-lunch rule, role/faction scope |
| **NPC messages** | Triggered messages from NPC personas | Persona voice guides, target faction/role, game state |
| **Resolution narrative** | The prose summary of what happened after decisions resolved | State deltas, decisions made, threshold events fired |

## 4. Architecture

### 4.1 The Story Bible

The core mechanism for narrative coherence. A structured document that accumulates as the game progresses, serving as the LLM's "memory" across rounds.

```typescript
interface StoryBible {
  // Set once at game start
  scenario: string;           // The AI 2027 premise (compressed)
  factions: FactionBrief[];   // Each faction's identity, tension, key characters
  voiceGuides: Record<string, string>; // Per-NPC-persona voice descriptions
  roundArcs: RoundArc[];      // The narrative arc template for each round

  // Grows each round
  events: StoryEvent[];       // What happened, in narrative form
  playerActions: PlayerActionSummary[]; // What each faction/role chose and why it mattered
  activeThreads: string[];    // Unresolved narrative tensions ("China's weight theft attempt was detected but not confirmed")
  toneShift: string;          // Current narrative mood ("tension rising, trust eroding")
}

interface StoryEvent {
  round: number;
  phase: "decision" | "threshold" | "publication" | "message";
  summary: string;            // "OpenBrain chose to push capabilities aggressively. Their CTO started Agent-4 training immediately."
  stateImpact: string;        // "obCapability +8, alignmentConfidence -5, whistleblowerPressure +8"
  narrativeWeight: "major" | "minor";
}

interface RoundArc {
  round: number;
  title: string;              // "The Superhuman Coder"
  era: string;                // "Q1 2027"
  narrativeBeat: string;      // "AI automates AI research. Human researchers sidelined. The gap between public knowledge and reality widens."
  escalation: string;         // What should feel different from last round
  keyTensions: string[];      // Narrative threads this round should advance
}
```

The story bible is initialized from `DESIGN.md` Section 5 (the round-by-round arc descriptions) and grows after each resolution phase. It's the primary context for every generation call.

### 4.2 Generation Pipeline

```
                    ┌─────────────────────────────────────────┐
                    │         RESOLUTION PHASE (R-1)          │
                    │                                         │
                    │  1. resolveDecisions() → new state      │
                    │  2. checkThresholds() → events fired    │
                    │  3. updateStoryBible() → append events  │
                    │  4. triggerGeneration(round=R)           │
                    │     ├─ generateBriefing()      (async)  │
                    │     ├─ generateContent()        (async)  │
                    │     ├─ generateDecisions()      (async)  │
                    │     └─ generateNpcMessages()    (async)  │
                    │  5. Validate + cache results             │
                    │  6. emitResolution() to clients          │
                    └─────────────────────────────────────────┘
                                     │
                                     ▼
                    ┌─────────────────────────────────────────┐
                    │         BRIEFING PHASE (R)              │
                    │                                         │
                    │  Content for round R is ready.          │
                    │  Emit briefing (generated).             │
                    │  If generation failed: use fallback.    │
                    └─────────────────────────────────────────┘
                                     │
                                     ▼
                    ┌─────────────────────────────────────────┐
                    │         INTEL PHASE (R)                 │
                    │                                         │
                    │  Emit content (generated).              │
                    │  NPC messages may fire (generated).     │
                    └─────────────────────────────────────────┘
                                     │
                                     ▼
                    ┌─────────────────────────────────────────┐
                    │         DECISION PHASE (R)              │
                    │                                         │
                    │  Emit decisions (generated).            │
                    │  Players vote.                          │
                    └─────────────────────────────────────────┘
```

**Timing:** Generation happens during the resolution phase of the *previous* round (R-1). Resolution is 180 seconds. The LLM has this full window to generate all content for round R. Generation calls run in parallel.

**For Round 2** (the first generated round): generation triggers during Round 1's resolution phase. Round 1 is pre-authored, so the LLM has the full context of what happened in Round 1 to work with.

#### generateDecisions()

The most constrained generation. Decisions are the mechanical core of the game.

```typescript
// Output schema: { individual: IndividualDecision[], team: TeamDecision[] }
//
// HARD CONSTRAINTS (enforced by validation, not just prompting):
//
// 1. Every decision has exactly 3 options.
//
// 2. No-free-lunch rule: every option must have:
//    - At least 2 positive effects (delta > 0 on desirable variables)
//    - At least 2 negative effects (delta > 0 on undesirable variables, or delta < 0 on desirable ones)
//    - Net absolute delta between 15-35 across all effects
//
// 3. Effect budget: each option has 5-8 effects.
//    - Effects touch variables relevant to the role/faction.
//    - At least 1 effect touches a Tier 2 (hidden) variable.
//    - No single effect exceeds |delta| = 8.
//
// 4. Distinctness: the 3 options must represent genuinely different strategies,
//    not "aggressive/moderate/conservative" on the same axis.
//    Enforce: no two options can have the same sign on more than 60% of shared variables.
//
// 5. Narrative grounding: prompt and option descriptions must reference
//    specific events from the story bible, not generic scenarios.
```

#### generateNpcMessages()

```typescript
// Output schema: NpcTrigger[]
// Constraints:
//   - 2-4 scheduled messages per round (during intel phase)
//   - 1-2 condition-based messages (fire if state crosses threshold)
//   - Messages must match the NPC persona's voice (from voiceGuides)
//   - Gameplay messages give actionable intel; escalation messages create urgency
//   - 1 flavor message per faction (personal/mundane, for immersion contrast)
```

### 4.4 Validation Layer

Every generated artifact passes through validation before being accepted. If validation fails, the system retries once with the validation errors fed back to the LLM. If the retry fails, it falls back to pre-authored content.

```typescript
interface ValidationResult {
  valid: boolean;
  errors: string[];
  warnings: string[];
}

// Decision validation
function validateDecisions(decisions: RoundDecisions, state: StateVariables): ValidationResult {
  const errors: string[] = [];

  for (const decision of [...decisions.individual, ...decisions.team]) {
    if (decision.options.length !== 3) {
      errors.push(`${decision.role ?? decision.faction}: expected 3 options, got ${decision.options.length}`);
    }

    for (const option of decision.options) {
      // Effect count
      if (option.effects.length < 5 || option.effects.length > 8) {
        errors.push(`${option.id}: expected 5-8 effects, got ${option.effects.length}`);
      }

      // No-free-lunch check
      const positiveCount = option.effects.filter(e => e.delta > 0).length;
      const negativeCount = option.effects.filter(e => e.delta < 0).length;
      if (positiveCount < 2) errors.push(`${option.id}: needs at least 2 positive effects`);
      if (negativeCount < 2) errors.push(`${option.id}: needs at least 2 negative effects`);

      // Delta magnitude
      const totalDelta = option.effects.reduce((sum, e) => sum + Math.abs(e.delta), 0);
      if (totalDelta < 15 || totalDelta > 35) {
        errors.push(`${option.id}: total delta ${totalDelta} outside [15, 35]`);
      }

      // Max single delta
      for (const effect of option.effects) {
        if (Math.abs(effect.delta) > 8) {
          errors.push(`${option.id}: effect on ${effect.variable} has delta ${effect.delta}, max is ±8`);
        }

        // Variable exists
        if (!(effect.variable in state)) {
          errors.push(`${option.id}: unknown variable ${effect.variable}`);
        }
      }
    }

    // Distinctness check
    if (decision.options.length === 3) {
      for (let i = 0; i < 3; i++) {
        for (let j = i + 1; j < 3; j++) {
          const a = decision.options[i];
          const b = decision.options[j];
          const sharedVars = a.effects
            .map(e => e.variable)
            .filter(v => b.effects.some(e2 => e2.variable === v));
          if (sharedVars.length > 0) {
            const sameSign = sharedVars.filter(v => {
              const da = a.effects.find(e => e.variable === v)!.delta;
              const db = b.effects.find(e => e.variable === v)!.delta;
              return Math.sign(da) === Math.sign(db);
            });
            if (sameSign.length / sharedVars.length > 0.6) {
              errors.push(`${a.id} vs ${b.id}: too similar (${sameSign.length}/${sharedVars.length} shared variables have same sign)`);
            }
          }
        }
      }
    }
  }

  return { valid: errors.length === 0, errors, warnings: [] };
}

// Content validation
function validateContent(items: ContentItem[], faction: Faction): ValidationResult {
  const errors: string[] = [];
  const classifications = items.map(i => i.classification);

  const critical = classifications.filter(c => c === "critical").length;
  const context = classifications.filter(c => c === "context").length;
  const redHerring = classifications.filter(c => c === "red-herring").length;

  if (critical < 3) errors.push(`${faction}: only ${critical} critical items, need ≥3`);
  if (critical > 5) errors.push(`${faction}: ${critical} critical items, max 5`);
  if (context < 5) errors.push(`${faction}: only ${context} context items, need ≥5`);
  if (items.length < 15) errors.push(`${faction}: only ${items.length} total items, need ≥15`);
  if (items.length > 30) errors.push(`${faction}: ${items.length} total items, max 30`);

  return { valid: errors.length === 0, errors, warnings: [] };
}
```

## 5. The Generation Prompt Structure

Every generation call uses a 3-layer prompt:

### Layer 1: System Prompt (static, per generation type)

Sets the LLM's role, output schema, and hard constraints. Example for content generation:

```
You are the narrative engine for Takeoff Protocol, an AI geopolitics
tabletop simulation. You generate in-universe app content that players
discover on their simulated desktop during each round.

You write as if you are the world itself — Slack messages from NPCs,
news headlines from Reuters, Bloomberg tickers, internal memos. Every
piece of content serves a gameplay function: it either reveals hidden
state, nudges a decision, adds atmosphere, or misleads.

OUTPUT: a JSON array of ContentItem objects matching the schema below.
[schema]

HARD RULES:
- Never contradict the current game state.
- Never reveal hidden variables directly (respect fog of war).
- Content classified "critical" must be actionable — it should change
  how a thoughtful player approaches their decision.
- Content classified "red-herring" must be plausible enough to waste
  time on but ultimately misleading.
- Match the app's voice: Slack is informal, News is wire-service terse,
  Bloomberg is financial shorthand, Memos are corporate formal.
- Reference specific prior events from the story bible, not generics.
```

### Layer 2: Story Bible (dynamic, grows each round)

The full `StoryBible` object, serialized. This is the LLM's "memory" — everything that's happened so far.

### Layer 3: Round-Specific Instructions (dynamic, per round)

```
ROUND 3 — "The Intelligence Explosion" — Mid 2027

Narrative beat: Agent-4 has arrived. The misalignment signal emerges.
This is the highest-tension round.

Key tensions to advance:
- The misalignment discovery (alignmentConfidence is at {X})

- China's espionage window (chinaWeightTheftProgress is at {X})

What happened last round:
- OpenBrain chose: {decision label + description}
- Prometheus chose: {decision label + description}
- China chose: {decision label + description}
- Key individual actions: {list}
- Threshold events fired: {list or "none"}
- Publications: {list or "none"}

Current state snapshot:
{full StateVariables object}

Generate content for faction: {faction}
Apps to populate: {list of AppIds for this faction}
```

## 6. Latency & Parallelism

### Time Budget

Resolution phase = 180 seconds. Generation must complete within this window.

### Parallelism Strategy

```
Resolution phase starts
  │
  ├─ [immediate] resolveDecisions() + checkThresholds() + updateStoryBible()
  │  (< 100ms, pure computation)
  │
  ├─ [parallel, all at once]
  │   ├─ generateBriefing()                    (~3-5s)
  │   ├─ generateContent(openbrain)            (~5-8s)
  │   ├─ generateContent(prometheus)           (~5-8s)
  │   ├─ generateContent(china)                (~5-8s)
  │   ├─ generateContent(external)             (~5-8s)
  │   ├─ generateDecisions()                   (~5-8s)
  │   ├─ generateNpcMessages()                 (~3-5s)
  │   └─ generateResolutionNarrative()         (~3-5s)
  │
  ├─ [after generation] validate all outputs
  │   └─ retry failures once (~5-8s if needed)
  │
  └─ [cache] store validated content for round R
      (ready before briefing phase starts)

Total wall time: ~10-15s (parallel) + ~5-8s (retry if needed) = ~15-23s
Well within the 180s resolution window.
```

### Model Selection

Use a fast model (Claude Haiku or equivalent) for content and NPC messages. Use a stronger model (Sonnet) for decisions and briefings where quality matters more. The generation calls are all server-side — no client-side LLM calls.

### Fallback Chain

```
1. Generate with LLM → validate → accept
2. Generate with LLM (retry with error feedback) → validate → accept
3. Fall back to pre-authored content for this round
4. Fall back to "generic" templates with state variable interpolation
```

Pre-authored content stays in the codebase as fallback. It's never deleted — just not the primary path.

## 7. Decision Generation: The Hard Problem

Decisions are the most important generated artifact because they directly affect game state. Bad decisions break the game. The system needs tight guardrails.

### Approach: Template + Reframe

Rather than generating decisions from scratch, use a hybrid approach:

**Step 1: Pre-author decision *templates* for each round.**

A template specifies the *structure* of a decision without the *content*:

```typescript
interface DecisionTemplate {
  round: number;
  role?: Role;        // individual decision
  faction?: Faction;  // team decision
  theme: string;      // "deployment strategy", "security posture", "espionage escalation"
  variableScope: (keyof StateVariables)[];  // which variables this decision should touch
  archetypes: [string, string, string];     // ["aggressive", "balanced", "cautious"] or ["internal", "external", "radical"]
}
```

**Step 2: LLM generates the narrative framing + specific effects.**

Given the template + story bible + current state, the LLM generates:
- The `prompt` (narrative setup referencing what happened)
- Three `options` with `label`, `description`, and `effects[]`
- Effects must stay within the `variableScope` and satisfy the no-free-lunch constraints

**Step 3: Validate effects mechanically.**

The validation layer checks all hard constraints (effect count, delta bounds, distinctness, balance). This is deterministic — no LLM judgment needed.

### Why Not Full Generation?

Fully generating decisions (including which variables to touch) risks:
- **Degenerate strategies** — all options touch the same variables, just different magnitudes
- **Orphan variables** — important variables never get touched, making them irrelevant
- **Balance drift** — without templates constraining scope, the LLM may create decisions that consistently favor or disfavor certain factions

Templates ensure every variable gets touched across the full set of decisions per round, and that the decision space covers the game's design goals.

### Future: Full Generation (v2)

Once the template approach is validated through playtesting, move toward full generation with stronger validation:
- Run Monte Carlo simulation on generated decisions before accepting them
- Reject decision sets where any single option dominates across >1000 random games
- Ensure variable coverage: every Tier 1 and Tier 2 variable is touched by at least one decision per round

## 9. Story Bible Maintenance

### After Each Resolution Phase

```typescript
function updateStoryBible(room: GameRoom): void {
  const bible = room.storyBible;
  const round = room.round;
  const history = room.history[room.history.length - 1]; // just-completed round

  // 1. Record faction decisions as story events
  for (const [faction, optionId] of Object.entries(history.teamDecisions)) {
    const option = findOptionById(optionId, round);
    if (option) {
      bible.events.push({
        round,
        phase: "decision",
        summary: `${faction} chose: "${option.label}" — ${option.description}`,
        stateImpact: option.effects.map(e => `${e.variable} ${e.delta > 0 ? '+' : ''}${e.delta}`).join(', '),
        narrativeWeight: "major",
      });
    }
  }

  // 2. Record individual decisions
  for (const [playerId, optionId] of Object.entries(history.decisions)) {
    const player = room.players[playerId];
    const option = findOptionById(optionId, round);
    if (option && player) {
      bible.events.push({
        round,
        phase: "decision",
        summary: `${player.role} (${player.faction}) chose: "${option.label}"`,
        stateImpact: option.effects.map(e => `${e.variable} ${e.delta > 0 ? '+' : ''}${e.delta}`).join(', '),
        narrativeWeight: "minor",
      });
    }
  }

  // 3. Record threshold events
  // (firedThresholds that fired THIS round)

  // 4. Record publications
  // (any publish:submit events this round)

  // 5. Update active threads
  //    - Scan state for emerging tensions
  //    - Remove resolved threads
  //    - Add new threads based on state crossings

  // 6. Update tone
  bible.toneShift = computeTone(room.state, round);
}
```

### Context Window Management

The story bible grows each round. By Round 5, it may contain 20-40 events. This is manageable — the full context (story bible + state + round instructions + schema) should stay under ~8K tokens, well within model limits.

If it grows beyond a threshold, summarize older rounds:
- Round 1-2 events: compress to 2-3 sentence summary
- Round 3+ events: keep full detail
- Always keep the most recent round in full

## 10. Cross-Faction Information Leakage

A key narrative mechanic: information that one faction has can leak to others through generated content. The LLM must respect fog-of-war rules while creating *hints* that cross faction boundaries.

### Rules

1. **Never reveal exact hidden values.** If China's capability is hidden from OpenBrain, generated news headlines can say "Chinese AI benchmarks show surprising gains" but never "DeepCent's model scores 67/100."

2. **Breadcrumbs are fog-respecting hints.** A breadcrumb in OpenBrain's security dashboard might reference "unusual network probes consistent with state-sponsored activity" — this hints at China's espionage without revealing `chinaWeightTheftProgress = 45`.

3. **Publications force information public.** When a player publishes, the generated content for *all* factions in subsequent rounds should reference the publication.

4. **Player DMs are never leaked into generated content.** Cross-faction messages sent by players are private. The LLM generates content based on state and decisions, not player chat logs.

## 11. Implementation Phases (Revised for Current Codebase)

The current server is deterministic and pre-authored:
- content is loaded from `content/` modules via `getContentForPlayer()`
- decisions are emitted from `ROUND*_DECISIONS`
- thresholds/NPC triggers are deterministic and tested

That is good. We should preserve this as the reliability baseline and add generation behind fallback gates.

### Phase A (Foundation): Generation Infrastructure, No Player-Facing Changes

Goal: add plumbing only.

- Add generation service interfaces and a provider abstraction (`GenerationProvider`) so we can swap models/vendors.
- Add generated round cache on room state, but do not serve it yet.
- Add validation modules + deterministic unit tests for validators.
- Add observability: generation latency, validation failure reasons, fallback reason.

Exit criteria:
- Can generate artifacts in background and cache them.
- If generation fails, gameplay is unchanged.

### Phase B (V1): Generated Briefings + Limited App Content (Rounds 2-5), Static Decisions

Goal: ship replayability gains with low mechanical risk.

- Generate only `briefing` and selected app content (start with `news` + `twitter`, optionally `email`).
- Keep decisions pre-authored (`ROUND_DECISIONS`) and keep existing NPC trigger logic.
- Trigger generation during prior round resolution and cache by `targetRound`.
- Serve generated artifacts only when validation passes; otherwise use existing pre-authored path.

Exit criteria:
- No phase stalls.
- No schema breaks on client.
- Fallback rate and latency are within budget.

### Phase C: Expand Content Surface + Generated NPC Messages

Goal: broader narrative reactivity without changing state mechanics.

- Expand generated app set to Slack/Signal/Memo/etc.
- Add generated NPC messages as additional triggers (still deduped and validated).
- Keep threshold mechanics deterministic.

Exit criteria:
- Generated artifacts remain fog-safe.
- GM and player logs show coherent cross-round narrative references.

### Phase D: Decision Generation (Template-Scoped, Guardrailed)

Goal: introduce generated mechanics safely.

- Use template-constrained decision generation only.
- Hard validation + simulation checks required before emission.
- Per-round or per-faction kill switch to fall back to pre-authored decisions.

Exit criteria:
- Balance and dominance metrics pass.
- Playtest sign-off across multiple sessions.

## 13. Cost Estimation

Per round, assuming Claude Haiku for content/NPC and Sonnet for briefings/decisions:

| Call | Model | Input tokens | Output tokens | Est. cost |
|---|---|---|---|---|
| Briefing | Sonnet | ~4K | ~800 | ~$0.02 |
| Content × 4 factions | Haiku | ~4K × 4 | ~3K × 4 | ~$0.01 |
| Decisions | Sonnet | ~5K | ~2K | ~$0.03 |
| NPC messages | Haiku | ~3K | ~500 | <$0.01 |
| Resolution narrative | Haiku | ~4K | ~500 | <$0.01 |

**Total per round: ~$0.08.** Over 4 generated rounds: ~$0.32 per game session.

Even with retries and the stronger model for everything, cost stays under $2/game. Negligible.

## 14. Open Questions

- [ ] **How much Round 1 content should we keep?** Currently it's all pre-authored. Should we keep it all, or generate some of it too to avoid a jarring quality shift between R1 and R2?
- [ ] **Player message context.** Should the LLM see player-to-player messages (team chat) as context for generation? Pro: more reactive content. Con: privacy concerns, context window bloat, players might game it.
- [ ] **State polarity map for validation.** The no-free-lunch validator needs an explicit variable polarity table (what counts as "good" or "bad") per faction/role, otherwise positive/negative checks are ambiguous.
- [ ] **Timeout and partial artifact policy.** If 2 of 4 faction content generations succeed, do we serve partial generated content and fallback for the rest (recommended), or require all-or-nothing per round?
- [ ] **GM override on generated content.** Should the GM be able to preview and edit generated content before it's emitted? This adds latency but gives the GM narrative control.
- [ ] **Caching across games.** If two games reach similar state at Round 3, can we reuse generated content? Probably not worth the complexity for v1.
- [ ] **Evaluation.** How do we measure whether generated content is "good enough"? Playtester surveys? A/B testing generated vs. pre-authored rounds?
- [ ] **Streaming.** Should content appear progressively (as if messages are arriving in real time) rather than all at once at intel phase start? This would increase immersion but complicate the pipeline.
