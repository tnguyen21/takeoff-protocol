# Generative Content System

## Design Document (v0.2)

*Updated 2026-03-19. Reflects current implementation state.*

---

## 1. The Problem

The game has ~375 hand-authored content items across 5 rounds, plus 5 sets of pre-written decisions. This content is static — the same Slack messages, news headlines, and decision prompts appear regardless of what players did. Content can only gate on numeric thresholds, not reference *what happened* or *who did what*.

After Round 1 (pre-authored "inciting incident"), all subsequent content — app items, briefings, NPC messages, and decisions — is generated by an LLM, reactive to game state and player choices.

---

## 2. Design Principles

**Skeleton + flesh.** The narrative skeleton (5-round arc, escalation curve, key story beats, threshold events) stays pre-authored. The LLM generates the flesh: specific messages, headlines, memos, and decision framings that react to what actually happened.

**State is the source of truth.** The LLM never invents state. It reads `StateVariables`, `RoundHistory[]`, and `firedThresholds`. It generates content that *reflects* this state, not content that *contradicts* it.

**Structured output, not free text.** The LLM outputs typed JSON matching existing schemas. The server validates and injects into the existing pipeline. Client code doesn't change.

**No free lunches, even in generation.** Generated decisions must have genuine tradeoffs. Every option must advance at least one variable while degrading at least one other. Enforced structurally via validation.

**No fallback by design.** If generation fails or the API is down, content is simply missing. Players are notified via a macOS-style degradation toast. There is no pre-authored fallback for rounds 2-5.

## 3. What Gets Generated vs. What Stays Fixed

### Fixed (pre-authored, all rounds)

| Element | Why it stays fixed |
|---|---|
| **Round 1 content** (all apps) | Inciting incident must be tightly controlled. |
| **State variable definitions** | 33 variables, ranges, clamping rules. |
| **Fog of war rules** | Structural, not narrative. |
| **Threshold events** | 8 once-per-game mechanical events. |
| **Ending arc resolvers** | 9 ending arcs — the scoring function. |
| **Phase structure & timing** | 5 phases per round. |
| **Activity penalties** | Primary app requirements per role. |

### Generated (Rounds 2-5)

| Element | Status |
|---|---|
| **Briefings** — `common` text + `factionVariants` per round | Implemented |
| **App content** — `ContentItem[]` per faction per app (all apps) | Implemented |
| **Decisions** — template-constrained, individual + team | Implemented |
| **NPC messages** — triggered messages from NPC personas | Implemented |
| **Resolution narrative** — prose summary of round outcomes | **Not yet implemented** |

The full generation system is implemented in `packages/server/src/generation/`:

| Module | File | Notes |
|--------|------|-------|

| NPC generation | `npc.ts` | Persona validation, post-processing filters |
| Validation | `validate.ts` | 5 validators + fog-safety heuristic |
| Caching | `cache.ts` | Per-round artifact + status storage (pending/ready/failed) |
| Orchestration | `orchestrator.ts` | Idempotent, never-throws, metrics, degradation toast via socket |
| Kill switches | `config.ts` | `GEN_ENABLED`, `GEN_BRIEFINGS_ENABLED`, `GEN_NPC_ENABLED`, `GEN_DECISIONS_ENABLED`, per-room GM toggle |

| Prompts | `prompts/` | 3-layer system prompts + faction/app voices + round arcs |

**Types** in `packages/shared/src/types.ts`: `StoryBible`, `GeneratedRoundArtifacts`, `GenerationStatus`, `StoryEvent`, `RoundArc`, `NpcTrigger`.

**GameRoom** fields: `storyBible`, `generatedRounds`, `generationStatus`, `generationEnabled`.

**Test coverage**: 13 test files, ~4,900 lines, covering all modules.

### Pipeline Flow

Generation triggers during resolution of round R-1 for round R (`game.ts`):

1. `resolveDecisions()` → new state
2. `checkThresholds()` → events fired
3. `updateStoryBible()` → append events
4. `triggerGeneration(room, round+1)` → async, non-blocking
5. Results cached in `room.generatedRounds[round]`

`emitBriefing()` and `emitContent()` read from the generation cache. If cache is empty, briefing shows placeholder text and content is empty array.

### Model Selection

| Artifact | Model | Configurable via |
|----------|-------|------------------|
| Briefing | Claude Sonnet | `GEN_BRIEFING_MODEL` |
| Content | Claude Haiku | `GEN_CONTENT_MODEL` |
| Decisions | Claude Sonnet | `GEN_DECISION_MODEL` |
| NPC messages | Claude Haiku | (uses content model) |

### Decision Generation Approach

Uses pre-authored **templates** that constrain which variables each decision touches:

```typescript
interface DecisionTemplate {
  round: number;
  role?: Role;
  faction?: Faction;
  theme: string;
  variableScope: (keyof StateVariables)[];
  archetypes: [string, string, string];
}
```

The LLM generates narrative framing + specific effects within the template's scope. Hard validation enforces: exactly 3 options, 5-8 effects per option, |delta| <= 8, no-free-lunch (>= 2 positive and >= 2 negative effects), distinctness (< 60% same-sign overlap on shared variables).

### Fog-Safety Validation

Heuristic scan of generated content for leaked hidden state values (`validate.ts`). Computes fog view per faction, searches item bodies for numeric values of hidden variables near relevant keywords. Non-blocking (warnings only).

## 5. Outstanding Work

### P1: Resolution Narrative Generation

The pipeline diagram originally included `generateResolutionNarrative()` but this was **never implemented**. The resolution phase currently uses a generic template, not LLM-generated prose.

**What it should do:**
- Generate a reactive prose summary of what happened after decisions resolved
- Reference specific decisions made, state deltas, threshold events fired
- Set the tone for the next round's briefing

**Inputs:** State deltas, decisions made, threshold events, story bible.

### P1: Cost Controls

**No cost controls exist.** ~14 API calls/round/room.

Missing:
- Concurrent request throttle (e.g., max 5 simultaneous Anthropic API calls)
- Per-room budget cap
- Daily spend cap
- Token usage logging from Anthropic response metadata

### P1: Live Playtesting & Quality Validation

The entire generation system has only been tested with MockProvider. Prompt quality, latency, and actual API costs are **untested with real API calls**.

Needed:
- Run playtests with `GEN_ENABLED=true` + real `ANTHROPIC_API_KEY`
- Measure actual latency and cost per round
- Validate narrative coherence across 5 rounds
- Tune prompts based on real output quality

### P2: GM Generation Visibility

The GM dashboard has no visibility into generation status (pending/ready/failed) or generated content preview. The `game:generation-status` socket event exists but isn't surfaced in the GM UI.

### P3: Decision Generation v2 — Full Generation

The current template approach constrains which variables each decision touches. A future v2 could fully generate decisions with stronger validation:
- Run Monte Carlo simulation on generated decisions before accepting them
- Reject decision sets where any single option dominates across >1000 random games
- Ensure variable coverage: every Tier 1 and Tier 2 variable touched per round

### P4: Streaming Content Delivery

Currently all content for a round arrives at once at intel phase start. Streaming would deliver content progressively (as if messages arriving in real time) for immersion. Would complicate the pipeline.

## 6. Cost Estimation

**These are design-phase estimates. Real API costs are untested.**

Per round, assuming Claude Haiku for content/NPC and Sonnet for briefings/decisions:

| Call | Model | Input tokens | Output tokens |
|---|---|---|---|
| Briefing | Sonnet | ~4K | ~800 |
| Content x 4 factions | Haiku | ~4K x 4 | ~3K x 4 |
| Decisions (~20 templates) | Sonnet | ~5K x 20 | ~2K x 20 |
| NPC messages | Haiku | ~3K | ~500 |

The original estimate of ~$0.08/round did not account for per-template decision generation calls. STATUS.md estimates ~$6.64/round. True cost will only be known after real API testing.

## 7. Open Questions

- [ ] **Player message context.** Player Slack messages are currently included in generation context. Should we also include DMs/team chats? Pro: more reactive. Con: privacy, context bloat, gaming.
- [ ] **GM override on generated content.** Preview and edit before emission? Adds latency but gives narrative control.
- [ ] **Evaluation.** How do we measure "good enough"? Playtester surveys? A/B testing?
- [ ] **Streaming.** Progressive content delivery during intel phase?

### Resolved

- **Round 1 content** — All pre-authored, by design.
- **Timeout and partial artifact policy** — Each faction/artifact fails independently; no all-or-nothing.
- **Caching across games** — Not implemented, not planned.
- **State polarity map** — Handled via template variableScope constraints.
- **Fallback strategy** — No pre-authored fallback by design; degradation toast notifies players.
